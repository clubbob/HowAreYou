# 프리미엄 1안 (안전형) 설계

> 보호대상자 무제한 등록 · 스마트 이상 감지 · 실시간 무활동 알림

---

## 1. 기능 요약

| 기능 | 설명 |
|------|------|
| **보호대상자 무제한 등록** | 프리미엄 보호자는 2명 초과 등록 가능 (이미 구현됨) |
| **스마트 이상 감지** | 6h / 9h / 12h 단계별 무활동 감지 |
| **실시간 무활동 알림** | 단계 도달 시 즉시 보호자에게 FCM 발송 |

---

## 2. subjects 문서 필드 (프리미엄용)

| 필드 | 타입 | 설명 |
|------|------|------|
| `lastMovementAt` | Timestamp \| null | 마지막 핸드폰 물리적 이동 시각 (Activity Recognition) |
| `lastInactivityAlertAt` | Timestamp \| null | 마지막 무활동 알림 발송 시각 |
| `lastInactivityAlertLevel` | number | 마지막 발송 단계 (0=미발송, 1=6h, 2=9h, 3=12h) |

**무활동 기준**: `lastMovementAt` 원칙. 없거나 센서 비정상 시에만 `lastResponseAt` 보조 사용

---

## 3. 리셋 조건 (필수)

**보호대상자가 안부를 남기면 반드시 다음 필드를 초기화해야 합니다.**

```
lastInactivityAlertAt = null (또는 필드 삭제)
lastInactivityAlertLevel = 0
```

### 리셋이 필요한 이유
- 이 초기화 로직이 없으면 다음 사이클에서 단계 계산이 꼬임
- 예: 12h 알림 발송 후 → 보호대상자가 기록 → 리셋 안 하면 → 다음 스캔에서 "이미 12h 알림 보냄"으로 잘못 판단 가능

### 리셋 수행 시점
1. **prompts 문서 생성 시** (onResponseCreated) — **서버가 최종 진실(source of truth)**
2. **prompts 문서 업데이트 시** (onResponseUpdated)
3. ~~Flutter mood_service~~ — **제거됨**: 서버 전용으로 통일 (Rules에서 클라 쓰기 차단)

### 정합성 우선순위 (필수 원칙)
| 구분 | 역할 |
|------|------|
| **서버(Cloud Functions)** | 최종 진실. 스케줄러가 읽는 값은 서버 기준. |
| **클라 리셋** | "즉시 UI/체감"용 보조. 서버 실패/지연되어도 다음 스케줄러는 서버 값을 읽으므로 안정. |

→ **현재**: 서버만 리셋 (클라 쓰기 Rules 차단). 안정성 최우선.

## 4. 스마트 이상 감지 단계

| 단계 | 무활동 시간 | 알림 문구 (예시) |
|------|-------------|------------------|
| 1 | 6시간 | 6시간째 안부가 없습니다. |
| 2 | 9시간 | 9시간째 안부가 없습니다. |
| 3 | 12시간 | 12시간째 안부가 없습니다. 확인이 필요합니다. |

### 발송 조건 (프리미엄 보호자만)

**기준 시각 (lastResponseAt 단독 기준 제거)**

```
baseAt = lastMovementAt ?? lastResponseAt
if (now - baseAt >= 단계시간) → 발송 후보
```

- `lastMovementAt` 원칙, 없거나 센서 비정상 시에만 `lastResponseAt` 보조
- 해당 보호대상자와 연결된 보호자 중 1명 이상이 프리미엄

### sendPremiumInactivityAlert 필수 안전장치 2개

**(A) 같은 단계 재발송 방지**
```
lastInactivityAlertLevel == 현재단계 이고
lastInactivityAlertAt가 최근(예: 3h 이내)면 스킵
```

**(B) 단계 역행 방지**
```
현재단계 <= lastInactivityAlertLevel 이면 스킵
```
→ 타임스탬프 꼬임 시 "2단계 발송 후 다시 1단계로 떨어지는" 구현 방지. "알림이 왔다갔다"하는 불신 방지.

## 5. Firestore Rules & Index

### Rules: lastInactivityAlert* 클라 쓰기 차단
- **안정성 최우선**: `lastInactivityAlertAt`, `lastInactivityAlertLevel` — **서버만 쓰기**
- subjects update 규칙에 `!('lastInactivityAlertAt' in request.resource.data) && !('lastInactivityAlertLevel' in request.resource.data)` 추가

### Index
- 스케줄러가 `lastResponseAt` 기준 정렬/필터를 걸면 인덱스 필요할 수 있음
- 쿼리 형태: `subjects.where('lastResponseAt', '<', threshold)` 등 → `lastResponseAt` 단일 필드 쿼리는 기본 인덱스로 충분
