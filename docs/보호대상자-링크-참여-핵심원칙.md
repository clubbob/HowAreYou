# 보호대상자 = 링크를 눌러 참여하는 사람 (핵심 원칙)

> 보호대상자는 ‘앱을 고르는 사람’이 아니라 **‘링크를 눌러 참여하는 사람’**이어야 한다.  
> 카톡/문자로 받은 링크를 클릭하는 순간, 이미 보호대상자로 설치·실행되는 것이 이 앱 UX의 핵심 원칙이다.

---

## 왜 “클릭하는 순간 보호대상자”인가

### 1. 보호대상자는 선택·설정하면 안 된다

부모(보호대상자) 입장에서:

- 모드 선택? ❌  
- 역할 이해? ❌  
- 계정 구조? ❌  

→ 이 순간 이미 피로해진다.

**보호대상자는 “눌렀더니 이미 정해져 있는 상태”여야 한다.**

### 2. “왜 깔아야 하는지” 설명 타이밍은 딱 한 번

그 타이밍 = **카톡/문자 메시지를 읽는 순간**.

그 다음은: 설치 → 실행 → 버튼 누르기.  
설명은 그 전에 끝나야 한다.

---

## 이상적인 전체 흐름

| 단계 | 담당 | 내용 |
|------|------|------|
| **1** | 보호자 | 앱 설치 → 보호자 모드 선택 *(현 구조 유지)* |
| **2** | 보호자 | 앱에서 **“초대 링크 생성”** → 링크 안에 보호자 ID, 관계 정보, 보호대상자 역할 포함 |
| **3** | 보호자 | 카톡/문자로 링크 전송. 앱이 **보낼 문장**까지 제공 (예: “요즘 연락 안 되면 내가 괜히 마음 쓰여서 그래. 이거 깔고 하루 한 번만 눌러주면 돼.”) |
| **4** | 보호대상자 | **링크 클릭** → Play Store 이동 → 설치 → 앱 실행 |
| **5** | 보호대상자 | **앱 실행 = 이미 보호대상자 모드**. 역할 선택 ❌, 설명 페이지 ❌, 로그인 먼저 ❌. 바로 “○○님이 괜히 마음 쓰이지 않게 하루 한 번만 알려주세요.” + 버튼 3개 |

---

## 이 구조의 장점

- **거부감 감소**: “감시/관리당한다”가 아니라 “저 사람이 걱정돼서 그렇구나”로 전달됨.
- **보호자 설명 부담 감소**: 말 꺼내기 어려움, 설명하다 싸움 → 앱이 말을 대신함.
- **이탈 포인트 제거**: “보호자/대상자 뭘 눌러?”가 사라지고, 한 버튼·한 행동으로 끝.

---

## 개발 관점 요약

- **딥링크 / Dynamic Link 필수**  
  링크에 `role=subject`(또는 target), `inviterId`(보호자 ID) 포함.
- **첫 실행 시 분기**
  - 링크 파라미터 있음 → **보호대상자 모드 고정**, 초대한 보호자와 연결.
  - 없음 → 기존처럼 **역할 선택 화면**.
- **결론**: 이 앱은 “설명 잘한 앱”이 아니라 **“설명할 필요가 없는 앱”**일 때 성공한다.

- [ ] 보호자 앱: “초대 링크 생성” 기능 (inviterId, role 등 포함)
- [ ] 보호자 앱: 링크 + 카톡/문자용 문장 세트 제공 (복사·공유)
- [ ] 딥링크 스킴 또는 Firebase Dynamic Links 설정
- [ ] 앱 첫 실행/冷 start 시 링크 파라미터 처리
- [ ] `role=subject` + `inviterId` 있을 때: 역할 선택 건너뛰고 보호대상자 화면으로 직행, 해당 보호자와 연결
- [ ] 보호대상자 첫 화면 문구: “○○님이 괜히 마음 쓰이지 않게 하루 한 번만 알려주세요.” + 버튼 3개
- [ ] (필요 시) 설치 후 첫 실행만 로그인 유도, 이후에는 링크로 들어온 보호대상자 모드 유지
