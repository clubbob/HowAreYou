# 프리미엄 서비스 — 실서비스 리스크 및 해결

> 운영에서 터질 수 있는 문제점 우선순위 정리

---

## P0 (치명적) — ✅ 해결됨

### 1. 움직임 갱신 시 알림 단계 미리셋

**문제**

- 리셋은 응답(안부) 작성/수정에서만 수행
- 기준은 `lastMovementAt`(물리적 이동)
- **시나리오**: 12h 알림 발송 → lastMovementAt만 갱신(응답 없음) → 다음 스캔에서 currentStage는 0~1인데, (B) "currentStage <= lastInactivityAlertLevel" 때문에 영원히 스킵

**해결 (구현 완료)**

- `onSubjectUpdated` 트리거: subjects 문서에서 `lastMovementAt` 변경 시 `lastInactivityAlertAt` 삭제, `lastInactivityAlertLevel = 0` 리셋
- "움직임이 생기면 새로운 사이클 시작"

---

## P1 (높음)

### 2. lastMovementAt 갱신이 SubjectModeScreen 진입에 묶임

**문제**

- `SubjectModeScreen` 진입 시 `start()`, 종료 시 `stop()`
- 앱을 잘 안 여는 보호대상자는 movement 데이터가 끊김
- "센서 미동작"을 "진짜 정지"로 오판 가능 → 6h 오탐으로 보호자 불신

**해결 방향**

- 부팅 후/앱 최초 1회 권한 동의 후 백그라운드 유지(Foreground service) 설계
- 또는 "센서 품질 알림"(권한/배터리 최적화 안내) 별도 제공

---

## P2 (중간)

### 3. STILL 제외 — 충전/책상 정상 정지 오탐

**문제**

- 충전 중, 책상 위, 수면 등 정상 정지가 많음
- 시간대/충전/예외 규칙 없음 → 오탐 확률 높음

**해결 방향**

- 기본 "야간 제외(예: 00~06)" 또는 "활동 시간대(예: 09~21)" 적용
- 6h는 특히 민감하므로 기본 예외 필요

---

### 4. sendPremiumInactivityAlert 쿼리 병합 — 누락 가능

**문제**

- `lastMovementAt`이 null인 문서는 movement 쿼리에 안 잡힘
- `lastResponseAt`만 오래된 경우 응답 쿼리에만 잡힘
- baseAt 계산 일관성 없으면 대상 누락/과잉 포함 가능

**해결**

- baseAt 계산 일관성 유지 (`baseAt = lastMovementAt ?? lastResponseAt`)
- null 케이스 명확히 처리

---

## P3 (성장 시)

### 5. 프리미엄 보호자 필터링 N+1

**문제**

- subject마다 pairedGuardianUids에서 프리미엄만 필터링
- 보호자 문서 개별 조회 시 스케줄러가 커질수록 읽기 비용 증가

**해결 방향**

- subject에 `premiumGuardianUids` 캐시 (Callable에서 갱신)
- 또는 guardian 프리미엄 여부를 배치로 한 번에 조회
