# 프리미엄 보호대상자 제한 - 점검 및 보완 사항

> 사업적으로 반드시 점검해야 할 3가지 + Firestore Rules 현황

---

## 1️⃣ 현재 상태: 클라이언트 기반 제한

### ✅ 구현됨
- `guardian_dashboard_screen.dart`: `currentSubjectCount >= 2`일 때 3번째 추가 차단
- 다이얼로그: "무료 플랜에서는 보호대상자 2명까지 등록할 수 있습니다. 3명 이상 추가를 원하시면 프리미엄 기능이 필요합니다."

### ❌ 미구현 (우회 가능)
- **Firestore Rules**: 보호대상자 수 제한 없음
- **Cloud Functions**: 없음 (functions 폴더 없음)
- **API**: 앱이 직접 Firestore에 쓰기

### 우회 시나리오
| 방법 | 가능 여부 |
|------|----------|
| 앱 수정 (클라이언트 체크 제거) | ✅ 가능 |
| Firestore SDK 직접 호출 | ✅ 가능 |
| 예전 버전 앱 | ✅ 가능 |
| API 직접 호출 | N/A (API 없음, 직접 Firestore) |

## 2️⃣ Firestore Rules 현황

### subjects/{subjectUid} write 규칙

```
allow write: if canRemoveSelfAsGuardian() || canAddSelfAsGuardian()
```

- **canAddSelfAsGuardian()**: 인증 사용자가 본인이 아닌 subjects 문서에 자신의 UID를 `pairedGuardianUids`에 추가하는 경우 허용
- **제한 없음**: "이 보호자가 이미 2명 이상 등록했는지" 체크 **없음**

### Firestore Rules 한계
- Rules는 **현재 쓰기 중인 문서**와 `get()`으로 가져온 **개별 문서**만 참조 가능
- "pairedGuardianUids에 X를 포함하는 subjects 문서가 몇 개인지" **쿼리 불가**
- 따라서 **Rules만으로는 보호대상자 수 제한 불가**

## 3️⃣ 서버 레벨 보완 방안

### 방안 A: Cloud Functions (Callable) ⭐ 권장

1. `addGuardianToSubject` Callable Function 생성
2. 클라이언트: `GuardianService.addMeAsGuardianToSubject` 대신 이 Function 호출
3. Function 내부:
   - Admin SDK로 `subjects.where('pairedGuardianUids', 'array-contains', guardianUid).get()` → count
   - `users/{guardianUid}`에서 `subscriptionStatus` 확인 (또는 별도 구독 서비스)
   - `count >= 2 && !isPremium` → throw
   - else → Admin SDK로 subjects 문서 업데이트

### 방안 B: users 문서에 subjectIds 유지

- `users/{guardianUid}.guardianSubjectIds: [id1, id2]` 저장
- 추가 시: 클라이언트가 users 문서도 업데이트 (subjectIds에 새 id 추가)
- Firestore Rules: `request.resource.data.guardianSubjectIds.size() <= 2` (무료) 또는 구독 필드 확인
- **문제**: 구독 상태를 Rules에서 어떻게 확인? users에 subscriptionStatus 있어야 함. 그리고 클라이언트가 users를 조작할 수 있으면 위조 가능.

### 방안 C: Cloud Functions (Trigger)

- `onDocumentWritten` subjects/{id} 트리거
- pairedGuardianUids 변경 시, 추가된 guardianUid에 대해 count 확인
- 3명 초과 && !isPremium → **롤백** (방금 추가한 guardian 제거)
- **단점**: 이미 쓰기 완료 후 롤백 → 복잡, 경쟁 조건 가능

**결론**: 방안 A (Callable Function)가 가장 안전하고 명확함.

## 4️⃣ 만료 처리 로직 점검

### 현재 subscription_service.dart
```dart
static SubscriptionState evaluate({...}) {
  return SubscriptionState._(
    phase: SubscriptionPhase.active,
    isRestricted: false,  // 항상 전체 기능
    message: '무료 이용 중',
  );
}
```
- **무료 전환 상태**: 구독 체크 비활성화, 항상 `isRestricted: false`

### 체크 포인트 (프리미엄 활성화 시)
| 항목 | 권장 |
|------|------|
| 시간 기준 | 서버 시간 (Firebase Admin Timestamp) |
| subscriptionExpiry null | 무료로 처리 |
| subscriptionStatus | enum: `free` \| `active` \| `expired` |
| 문자열 오타 | 상수/enum 사용 |

## 6️⃣ 요약

| 항목 | 현재 | 필요 조치 |
|------|------|----------|
| 클라이언트 제한 | ✅ 2명 차단 | 유지 |
| 서버 제한 | ❌ 없음 | **Cloud Functions Callable 추가** |
| Firestore Rules | subjects write에 수 제한 없음 | Rules만으로는 불가, Function 필요 |
| 구독 로직 | 무료 전환 (비활성) | 프리미엄 활성화 시 enum/서버 시간 적용 |
| 만료 시 UX | - | 기존 인원 유지, 추가만 차단 |

### 배포 전 필수
- **보호대상자 무제한**이 프리미엄 핵심이므로, 서버 레벨(Cloud Functions) 제한 **반드시** 구현 후 배포 권장.
